# 1. Архитектура кластера Kubernetes

## Что такое кластер

**Кластер** — это группа машин (физических или виртуальных), которые вместе запускают контейнеризованные приложения. Kubernetes управляет этими машинами как единым целым.

Машины делятся на два типа:

- **Control plane (управляющий слой)** — «мозг» кластера, принимает решения.
- **Worker nodes (рабочие узлы)** — «рабочие руки», на них реально запускаются приложения.

В minikube control-plane и worker обычно совмещены в одном узле.

## Компоненты Control Plane

| Компонент | Зачем |
|-----------|--------|
| **kube-apiserver** | Единая точка входа в кластер. Все запросы (`kubectl`, контроллеры, ноды) идут через него. REST API. |
| **etcd** | Распределённое key-value хранилище. Хранит **всё состояние** кластера (объекты, конфиги, секреты). |
| **kube-scheduler** | Решает, **на какой ноде** запустить новый Pod (по ресурсам, ограничениям, affinity). |
| **kube-controller-manager** | Набор контроллеров, которые «приводят кластер к желаемому состоянию» (например, контроллер ReplicaSet следит, чтобы было нужное число подов). |
| **cloud-controller-manager** | Интеграция с облаком (LB, тома, ноды). В minikube/локально обычно не нужен. |

## Компоненты Worker Node

| Компонент | Зачем |
|-----------|--------|
| **kubelet** | Агент на каждой ноде. Получает задания от API-сервера и запускает контейнеры через runtime. |
| **container runtime** | То, что реально запускает контейнеры: **containerd**, CRI-O и т.д. |
| **kube-proxy** | Сетевые правила (iptables/ipvs) для работы Service: маршрутизирует трафик к подам. |

## Поток «как создаётся под»

1. `kubectl apply -f pod.yaml` → запрос уходит в **kube-apiserver**.
2. apiserver сохраняет объект в **etcd**.
3. **scheduler** видит, что под не назначен на ноду, и выбирает её.
4. **kubelet** на этой ноде получает указание и через **container runtime** (containerd) запускает контейнер.
5. **kube-proxy** обновляет сетевые правила, если нужен Service.

## Проверка на своём кластере

```bash
kubectl get nodes
kubectl get pods -n kube-system
kubectl cluster-info
```

В `kube-system` вы увидите как раз компоненты control plane (api-server, scheduler, controller-manager, etcd, coredns и т.д.).
